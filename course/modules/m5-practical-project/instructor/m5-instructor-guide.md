## Week 11 Day 1：方案打磨

### 课前准备
- [ ] 确认各组选题（提前1天收集）
- [ ] 准备方案评审打分表
- [ ] 了解每组的企业背景和业务场景

### 课堂节奏

```
09:00-09:30  评审标准讲解（30min）
09:30-10:00  各组方案汇报（5min/组，30min）
10:00-10:15  休息
10:15-12:00  方案讨论与修改（105min）
14:00-16:00  方案修改（120min）
16:00-18:00  方案定稿，导师确认（120min）
```

### 讲师角色说明

```
讲师在M5的角色跟前面四周不同：

M1-M4：讲师是主讲人，输出知识
M5：讲师是辅导者，引导学员自己完成

核心原则：
- 不要帮学员做决定
- 不要直接给答案，要给方向
- 多提问，少给方案
- 鼓励试错，及时纠偏
```

### 评审标准讲解（30min）

```
今天开始M5实战项目，这是90天课程的最后一公里。

前面的学习都是「学」，从今天开始是「做」。
你们要用前面学到的所有能力，完成一个真实的AI落地项目。

评审看四个维度：

第一，项目完整度（30%）
核心功能能不能跑起来？
工作流是不是闭环的？
能不能演示？

第二，商业价值（25%）
解决了什么问题？
带来了什么收益？
能不能算清楚ROI？

第三，技术实现（20%）
工作流设计是否合理？
节点配置是否正确？
有没有考虑到异常情况？

第四，答辩表现（15%）
能不能把项目讲清楚？
能不能回答评委的问题？

第五，创新性（10%）
有没有什么独特的亮点？

今天的目标是：每个组确定最终选题，拿出完整的方案设计。
```

## Week 11 Day 2：方案评审

### 评审流程

```
上午：正式评审（9:00-12:00）

每组15分钟：
1. 方案汇报（5min）
2. 讲师提问（5min）
3. 讲师点评（5min）

下午：方案定稿（14:00-18:00）

任务：
1. 根据上午评审意见修改方案
2. 确认最终技术方案
3. 分解开发任务
4. 分配组内职责
```

### 评审打分表（讲师用）

| 评分维度 | 权重 | 优秀（90+） | 良好（80+） | 及格（70+） | 待改进（<70） |
|---------|------|------------|------------|------------|-------------|
| 场景真实 | 20% | 真实业务，需求清晰 | 真实业务，需求基本清晰 | 有一定虚构 | 场景模糊 |
| 方案完整 | 30% | 闭环完整，异常考虑 | 基本完整 | 有缺口 | 无法闭环 |
| ROI可行 | 20% | 回本<6月，测算清晰 | 回本<12月 | 回本较长 | 无法测算 |
| 技术可行 | 20% | 完全可实现 | 基本可实现 | 有难度 | 超出能力 |
| 创新性 | 10% | 有独特亮点 | 有一定创意 | 常规 | 无创新 |

### 讲师提问清单

```
每个组必问的3个问题：

1. 你们的核心工作流是什么？
   （考察是否清楚自己要做什么）

2. 如果AI回复错了，你们怎么处理？
   （考察是否有异常处理意识）

3. 这个项目上线后，每年能省多少钱？
   （考察是否有商业意识）
```

## Week 11 Day 3-5：开发冲刺

### 每日站会（每天9:00，15分钟）

```
讲师主持，每个组1分钟：
1. 昨天完成了什么？
2. 今天要做什么？
3. 有什么阻碍？

讲师作用：
- 记录阻碍
- 及时协调资源
- 调整计划（如果需要）
```

### Day 3：任务分解与开发启动

```
上午：任务分解（60min）

任务分解原则：
1. P0任务不超过3个
2. 每个任务要有明确的完成标准
3. 每个任务分配到具体的人

下午：开发启动（180min）

目标：Day 3下班前，P0任务完成50%
```

### Day 4：核心功能开发

```
目标：Day 4下班前，核心功能可跑通

讲师巡场要点：
- 检查每个组的工作流设计
- 发现明显的配置错误及时指出
- 鼓励学员互相帮助
- 控制时间，提醒组员不要在细节上纠结
```

### Day 5：第一轮迭代

```
上午：功能完善

讲师提示：
- P1任务尽量完成
- 开始准备测试数据
- 功能基本冻结（不再新增）

下午：内部演示

要求：
- 每组演示最新进展
- 发现的问题记录到飞书文档
- 制定Day 6修复计划
```

### 讲师巡场话术

```
发现问题时的沟通方式：

❌ 错误："你这个配置不对，应该这样改"
✅ 正确："你们遇到什么问题了吗？我看到这里可能需要看一下……"

❌ 错误："这个功能做不完就砍掉吧"
✅ 正确："如果只留一个核心功能，你们会选择做哪个？为什么？"

核心：引导学员自己做决策，而不是替他们做。
```

## Week 12 Day 1-3：测试优化

### Day 6：错误修复

```
核心原则：
- 功能冻结，不再新增功能
- 只修复bug和优化体验
- 为答辩做准备

讲师任务：
- 帮助学员定位问题
- 提供测试用例清单
- 控制修改范围，避免引入新bug
```

### Day 7-8：交叉测试

```
测试配对建议：
- 3-4人一组
- 每组测试相邻编号的组

测试检查清单：
□ 核心功能是否可用
□ 异常输入是否有合理反馈
□ 工作流是否闭环
□ 消息是否正确发送/接收
□ 性能是否可接受（响应时间<3秒）
□ 界面是否整洁
```

### 测试报告反馈

```
每组测试后，讲师汇总问题：

| 问题 | 提出组 | 被测组 | 严重程度 | 是否已修复 |
|------|--------|--------|---------|-----------|
| 消息丢失 | A组 | B组 | 高 | 是 |

鼓励被测组感谢提出问题的组，形成正向氛围。
```

## Week 12 Day 4-5：答辩路演

### Day 9：答辩准备

```
上午：PPT制作指导（60min）

PPT结构建议（5页就够了）：
1. 项目背景（1页）：做什么业务、解决什么问题
2. 解决方案（1页）：工作流设计图
3. 关键配置（1页）：核心节点配置
4. 落地效果（1页）：量化指标
5. 经验总结（1页）：踩坑+复用经验

下午：彩排（180min）

每组模拟答辩一遍：
- 严格计时15分钟
- 讲师扮演评委提问
- 其他组观摩
```

### Day 10：正式答辩

```
答辩流程（每组20分钟）

| 环节 | 时长 |
|------|------|
| 项目汇报 | 10min |
| 演示环节 | 3min |
| 评委提问 | 5min |
| 点评总结 | 2min |

评委提问可能方向：
- 技术细节：为什么这样设计？
- 商业价值：这个ROI怎么算出来的？
- 异常处理：如果XX情况发生怎么办？
- 扩展性：如果要加XX功能，改动大吗？
```

### 最终评分汇总

```
L4认证通过标准：
- 总分 ≥ 75分
- 每个维度 ≥ 60分

总分构成：
- 项目完整度 30%
- 商业价值 25%
- 技术实现 20%
- 答辩表现 15%
- 创新性 10%
```

### 评优颁奖

```
答辩结束后，宣布：

1. L4认证通过名单
2. 优秀项目奖（总分第1名）
3. 最佳创新奖
4. 最佳答辩奖

奖品建议：
- 优秀项目：OpenClaw高级功能解锁
- 最佳创新：案例入库署名权
- 最佳答辩：预备讲师资格
```

## 讲师资源包

### M5 每日巡场 Checklist

```
每日巡场必看：
□ 每个组的核心工作流是否能跑起来
□ 组内是否有明显分工不均（有人忙有人闲着）
□ 是否有组因为技术难题卡住超过2小时
□ 是否有组方向偏离需要拉回

每日结束前：
□ 收集各组今日进展记录
□ 标记明日重点关注组
□ 更新整体进度表
```

### 常见问题应对

| 问题 | 应对 |
|------|------|
| 组内搭便车 | 加重答辩表现评分中的个人贡献项 |
| 进度严重落后 | 协助砍需求，确保核心功能能演示 |
| 技术方案推翻 | 引导回到已验证的方案，不鼓励冒险 |
| 组内冲突 | 分别沟通，强调共同目标 |
| 企业导师缺席 | 提前联系企业PM，请假需补交书面反馈 |

### L4认证打分权重提醒

```
这个分数关系到学员能不能拿到L4认证。
评分要客观公正：
- 不要因为学员态度好就给高分
- 不要因为答辩紧张就扣分
- 严格按照评分标准打分

如果某组刚好在75分边缘：
- 看「项目完整度」是否≥60
- 「项目完整度」达标则通过
- 「项目完整度」不达标则不通过
```
