# M5 实战项目

> Week 11-12：真实场景从0到1落地，答辩验收

---

## 项目概述

### 核心目标

```
每个小组：落地1个真实业务场景
每位学员：掌握AI Builder完整落地能力
```

### 最终交付

| 交付物 | 说明 |
|-------|------|
| AI工作流 | 可运行的OpenClaw配置 |
| 项目Demo | 演示视频或在线演示 |
| 方案文档 | 完整的落地方案 |
| 复盘报告 | 项目总结和经验沉淀 |

---

## 项目节奏

```
Week 11 Day 1-3  → 方案打磨
Week 11 Day 4-5  → 开发冲刺
Week 12 Day 1-3  → 测试优化
Week 12 Day 4-5  → 答辩路演
```

---

## Phase 1：方案打磨（Week 11 Day 1-3）

### Day 1：方案确认

**上午（理论30min + 带练30min）**

```
内容：方案评审标准讲解

1. 方案完整性（40%）
   - 场景真实
   - 流程闭环
   - 技术可行

2. 商业价值（30%）
   - ROI测算
   - 可量化收益

3. 技术实现（30%）
   - 工作流设计
   - 集成方式
```

**下午：小组讨论**

```
任务：
1. 确认最终选题
2. 完善方案设计
3. 准备方案文档
4. 提交导师审核
```

### Day 2：方案评审

**上午：方案汇报**

```
每组15分钟

汇报内容：
1. 业务场景（3分钟）
2. 解决方案（5分钟）
3. 预期效果（2分钟）
4. 开发计划（3分钟）
5. 导师提问（2分钟）
```

**评审标准**

| 维度 | 权重 | 通过标准 |
|------|------|---------|
| 场景真实 | 20% | 来自真实业务 |
| 方案完整 | 30% | 流程闭环 |
| ROI可行 | 20% | 3个月内回本 |
| 技术可行 | 20% | 现有能力可实现 |
| 创新性 | 10% | 有独特价值 |

**下午：方案定稿**

```
任务：
1. 根据评审意见修改方案
2. 确认最终技术方案
3. 分解开发任务
4. 分配组内职责
```

### Day 3：开发启动

**上午：任务分解**

```
任务分解方法：

1. 列出所有功能点
2. 标注优先级（MoSCoW）
3. 评估工时
4. 分配负责人

示例：
┌────────────────────────────────────────────────┐
│ 任务清单                                        │
├────────────────────────────────────────────────┤
│ P0 必须完成                                      │
│ ├── 核心工作流配置                             │
│ ├── 消息通道集成                               │
│ └── 基础功能测试                               │
├────────────────────────────────────────────────┤
│ P1 应该完成                                      │
│ ├── 知识库配置                                 │
│ ├── 异常处理                                   │
│ └── 界面优化                                   │
├────────────────────────────────────────────────┤
│ P2 可以完成                                      │
│ ├── 高级功能                                   │
│ └── 性能优化                                   │
└────────────────────────────────────────────────┘
```

**下午：开发启动**

```
任务：
1. 创建工作流框架
2. 配置核心节点
3. 对接必要集成
4. 完成P0任务
```

---

## Phase 2：开发冲刺（Week 11 Day 4-5）

### Day 4：核心功能开发

**每日站会（每日开始）**

```
时间：9:00
时长：15分钟
参与：全组

内容：
1. 昨天完成了什么
2. 今天要做什么
3. 遇到什么阻碍

形式：简短高效，不超过15分钟
```

**开发任务**

```
P0任务（必须完成）：
- 核心工作流配置
- 消息通道集成
- 基础功能测试

目标：下班前核心功能可跑通
```

### Day 5：第一轮迭代

**上午：功能完善**

```
任务：
1. 完成P1任务
2. 完善异常处理
3. 开始测试
```

**下午：内部演示**

```
任务：
1. 组内演示最新进展
2. 发现问题记录
3. 制定Day 6优化计划
```

---

## Phase 3：测试优化（Week 12 Day 1-3）

### Day 6：错误修复

**上午：Bug修复**

```
任务优先级：
1. P0问题：核心功能bug（立即修）
2. P1问题：重要功能问题（今天修）
3. P2问题：次要问题（尽量修）
```

**下午：功能冻结**

```
规则：
- 功能冻结，不再新增功能
- 只修复bug和优化性能
- 为答辩做准备
```

### Day 7-8：交叉测试

**测试方法**

```
1. 小组交叉测试
   - A组测B组项目
   - B组测C组项目
   - 发现问题记录

2. 测试维度
   - 功能完整性
   - 异常处理
   - 性能表现
   - 用户体验
```

**测试报告模板**

```markdown
## 测试报告

### 测试人：
### 测试时间：
### 被测项目：

### 测试结果

| 功能 | 状态 | 问题描述 |
|------|------|---------|
| 功能1 | ✅/❌ | 无/问题描述 |
| 功能2 | ✅/❌ | 无/问题描述 |

### 问题清单

| 问题 | 严重程度 | 优先级 |
|------|---------|-------|
| 问题1 | 高/中/低 | P0/P1/P2 |

### 总体评价
```

## Phase 4：答辩路演（Week 12 Day 4-5）

### Day 9：答辩准备

**上午：PPT制作**

```
答辩PPT结构：

1. 项目背景（1页）
   - 业务场景
   - 核心痛点

2. 解决方案（2页）
   - 工作流设计
   - 关键配置

3. 落地效果（1页）
   - 量化指标
   - 收益分析

4. 经验总结（1页）
   - 踩坑点
   - 可复用经验
```

**下午：彩排**

```
任务：
1. 全组彩排答辩
2. 控制时间（15分钟）
3. 准备QA
```

### Day 10：正式答辩

**答辩流程**

```
时间安排：
- 8:30-9:00  设备调试
- 9:00-12:00  上午组答辩（5组）
- 12:00-13:30 午休
- 13:30-17:00 下午组答辩（5组）
- 17:00-18:00 评优颁奖

每组时间：
- 答辩：15分钟
- 提问：5分钟
- 点评：2分钟
```

**答辩评分表**

| 维度 | 权重 | 评分标准 |
|------|------|---------|
| 项目完整度 | 30% | 功能完整可演示 |
| 商业价值 | 25% | ROI明确可量化 |
| 技术实现 | 20% | 工作流设计合理 |
| 答辩表现 | 15% | 表达清晰、有条理 |
| 创新性 | 10% | 有独特价值点 |

**评委构成**

| 评委 | 数量 | 职责 |
|------|------|------|
| 内部导师 | 2人 | 技术评审 |
| 业务领导 | 1人 | 业务价值评审 |
| 产品经理 | 1人 | 需求分析评审 |
