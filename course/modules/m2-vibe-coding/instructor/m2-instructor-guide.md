## 核心理念：vibe coding 是能力阶梯

```
vibe coding 能力 = 工具使用 × 调试能力 × 工程化能力

这不是三个独立的东西，而是三个递进的能力阶段：

L1 工具使用：
→ 会用 Codex/Claude Code 生成代码
→ Prompt 写得好，代码质量高

L2 调试能力：
→ 代码出问题，能定位问题
→ 能引导 AI 正确修复

L3 工程化：
→ 代码可维护、可扩展、可协作
→ 能做前后端对接，能做代码重构

L4 综合：
→ 能独立完成完整项目
→ 综合运用三个能力

课程设计逻辑：
Week 3 → 打好工具使用基础
Week 4 → 建立调试能力
Week 5 → 提升工程化能力
Week 6 → 综合运用

每个能力的培养方式：
L1：多练，形成肌肉记忆
L2：在错误中学习，每次出错都是练习机会
L3：掌握规范，形成工程意识
```

## L1：工具使用（Week 3）

### Week 3 整体理念

```
L1 的目标：让学员熟练使用 Codex/Claude Code 生成代码。

核心心法：
Prompt 质量 = 代码质量
Prompt 写得好，代码一次对；Prompt 写得差，代码满是 bug。

两个关键能力：
1. 结构化 Prompt：把需求描述清楚
2. 错误预判：在 Prompt 里堵住可能的漏洞

L1 学完的标准：
学员能独立用 AI 生成一个完整的可运行页面/功能，
不需要频繁的修复和调试。
```

## M2-1 AI编程思维 + Codex入门（Week 3 Day 1）

### 课前准备
- [ ] 准备 Codex 账号并测试
- [ ] 准备「同一需求，不同 Prompt」对比案例
- [ ] 准备学员项目代码仓库

### 理论逐字稿（30min）

```
今天进入 M2。

M1 你们学了怎么用 AI Builder 搭建工作流——那是「连接」的能力。
M2 我们要学的是：用 AI 写代码——这是「实现」的能力。

先破除一个误解：
很多人觉得「会用 AI 写代码」就是「把需求扔给 AI」。
错了。

真实情况：
Prompt 写得好 → 代码一次对
Prompt 写得差 → 代码满是 bug，改来改去

所以 M2 的第一个核心能力：**Prompt 质量**。

给大家看两个例子：

例子1：低质量 Prompt
「写一个计算器」

例子2：高质量 Prompt
「用 HTML+CSS+JavaScript 写一个计算器，要求：
1. 支持加减乘除
2. 除以 0 显示"错误"
3. 输入非数字显示"输入错误"
4. 结果保留 2 位小数
5. 有清零功能」

两个 Prompt 出来的代码，质量差距巨大。

这就是 L1 的核心：
把 Prompt 写好，让代码一次做对。
```

### 跟练任务（90min）

```
任务1：体验 Prompt 质量差异（30min）

Prompt版本1：「写一个待办列表」

Prompt版本2：
「用 HTML+CSS+JavaScript 写一个待办列表，要求：
- 可以添加任务
- 可以删除任务
- 点击任务显示删除线（完成态）
- 数据保存在 localStorage
- 界面简洁现代」

对比：两版 Prompt 生成的代码质量差异

任务2：优化 Prompt（60min）

1. 用简单 Prompt 生成一个「用户登录页面」
2. 分析：生成的代码有哪些问题？
3. 重新写 Prompt，把问题堵住
4. 重新生成，对比质量
```

## M2-2 Codex进阶 + 错误预判（Week 3 Day 2）

### 核心理念

```
今天的核心：学会「错误预判」——在问题发生之前，在 Prompt 里堵住漏洞。

高手和新手的差距：
新手：出问题了再想办法
高手：在 Prompt 里就堵住问题

错误预判三步：
1. 想清楚 AI 可能在哪里出错
2. 在 Prompt 里明确约束
3. 让 AI 自己检查逻辑
```

### 理论逐字稿（30min）

```
昨天你们体验了 Prompt 质量的影响。
今天更进一步：学会「错误预判」。

什么是错误预判？
就是你想清楚 AI 可能在哪里出错，
然后在 Prompt 里把这些漏洞堵住。

举个例子：生成一个「表单验证」功能。

新手 Prompt：
「写一个表单验证」

高手 Prompt：
「用 JavaScript 写一个表单验证，要求：
1. 用户名不能为空，提示"用户名不能为空"
2. 密码至少 6 位，不满足提示"密码至少6位"
3. 邮箱格式验证，格式错误提示"邮箱格式不正确"
4. 提交时验证，有错误不提交
5. 错误信息显示在输入框下方」

区别在哪？
新手 Prompt：AI 自由发挥，可能遗漏边界情况
高手 Prompt：把可能的错误路径都堵住了

这就是「错误预判」。

错误预判三步法：
第一步：想清楚 AI 可能在哪里出错（边界条件、异常输入）
第二步：在 Prompt 里明确约束（把防错逻辑写进去）
第三步：让 AI 自己检查（写完后检查有没有遗漏）
```

### 跟练任务（90min）

```
任务：错误预判练习

场景：生成一个「文件上传」功能

第一步（30min）：先用简单 Prompt，看 AI 会出什么问题
Prompt：「写一个文件上传页面」
记录：生成的代码有哪些问题？

第二步（30min）：做错误预判，写高质量 Prompt
想清楚：文件上传可能有哪些问题？
- 文件类型限制？
- 文件大小限制？
- 空文件？
- 上传失败？

把这些都写进 Prompt：
「用 HTML+JavaScript 写一个文件上传页面，要求：
1. 只允许上传图片（jpg/png/gif）
2. 文件大小不超过 5MB，超过提示"文件过大"
3. 空文件提交提示"请选择文件"
4. 上传中显示进度条
5. 上传成功显示图片预览」

第三步（30min）：对比两版 Prompt 的输出质量
记录：高质量 Prompt 是否避开了之前的问题？
```

## L2：调试能力（Week 4）

### Week 4 整体理念

```
L2 的目标：建立调试能力——代码出了问题，知道怎么定位，怎么让 AI 正确修复。

核心心法：
调试不是「改代码」，而是「读懂报错 → 定位问题 → 引导 AI 修复」。

调试的常见错误：
1. 不读报错信息，直接让 AI 修 → AI 乱猜
2. 一次让 AI 修太多地方 → 改乱了
3. 修完不测试 → 表面修好了，实际还有问题

正确做法：
读懂报错 → 一次修一个问题 → 修完测试 → 再修下一个

L2 学完的标准：
学员遇到代码报错，能：
1. 读懂报错信息
2. 定位问题所在
3. 引导 AI 正确修复
4. 验证修复结果
```

## M2-3 代码调试：定位问题（Week 4 Day 1）

### 理论逐字稿（30min）

```
今天学调试。

调试的第一步不是「改」，而是「读」。

报错信息就是 AI 给你的线索。
很多人看到报错就慌了，把报错直接扔给 AI 说「帮我修」。

更好的做法：
第一步：先自己读懂报错（这一步可以让 AI 帮你）
第二步：再让 AI 帮你修

让 AI 帮你读报错：
「这个报错是什么意思？问题出在代码的第几行？是什么原因导致的？」

让 AI 帮你修：
「问题是：[描述具体问题]，请修复并说明改了哪里。」

区别：
前者是「让 AI 分析问题」
后者是「让 AI 解决问题并让你理解」
两者顺序不能乱。
```

### 跟练任务（90min）

```
任务：给学员一个有 bug 的代码，让 AI 分析问题

给一段有问题的代码（比如：除零、数组越界、类型错误）

1. 让学员把报错信息发给 AI：
「这个报错是什么意思？问题在哪？」

2. 让学员根据 AI 分析，明确说出：
「问题是：XXX」

3. 让学员引导 AI 修复：
「问题是 XXX，请修复，并说明改了哪里、为什么改、原来为什么错」

4. 修完测试，确认修好

5. 再让 AI 检查：
「这段代码还有没有类似问题？」
```

## M2-4 代码调试：修复问题（Week 4 Day 2）

### 理论逐字稿（30min）

```
昨天学了定位问题，今天学修复问题。

修复有一个铁律：**一次只修一个问题**。

为什么？
因为一次修太多，改动的代码太多，你不知道哪个修好了哪个。
而且如果出了新问题，你不知道是哪个改动引的。

正确节奏：
修一个问题 → 测试 → 确认修好 → 再修下一个

还有一条铁律：**修完必须测试**。

很多人让 AI 修完就说「好了好了」，然后跑一遍发现还有问题。
修完一个，马上测，确认修好了，再让 AI 修下一个。
```

### 跟练任务（90min）

```
任务：给一个有 6 个 bug 的计算器

要求：
1. 一次只让 AI 修一个问题
2. 修完马上测试，确认修好
3. 再让 AI 修下一个
4. 记录每个 bug 的修复过程

禁止：
- 让 AI 一次修多个 bug
- 不测试就跑下一个
```

## L3：工程化能力（Week 5）

### Week 5 整体理念

```
L3 的目标：让代码从「能跑」变成「跑得好」——可维护、可扩展、可协作。

工程化的三个维度：
1. 代码质量：结构清晰、有注释、可读性好
2. 项目结构：前后端分离、接口规范、可扩展
3. 可维护性：容易修改、容易 debug、容易协作

核心心法：
AI 写的代码通常能跑，但不一定「工程化」。
人要把「工程规范」翻译成 AI 能理解的语言。

L3 学完的标准：
学员能让 AI 生成「工程化」的代码，而不是「能用就行」的代码。
```

## M2-5 代码重构与代码质量（Week 5 Day 1）

### 理论逐字稿（30min）

```
今天学重构。

重构 = 不改变功能，让代码变得更好。

AI 写的代码通常能跑，但工程上可能有问题：
- 变量名随意
- 没有注释
- 代码结构乱
- 可能有性能问题

重构的两个方向：
1. 可读性重构：加注释、变量名清晰、结构整齐
2. 性能重构：去掉重复计算、优化循环、减少不必要的操作

重构的铁律：**每次小改，动完就测**。

因为重构很容易引入新 bug。
每次改一点点，测一遍，确保功能不变。
```

### 跟练任务（90min）

```
任务：给一段「能跑但很乱」的代码，让 AI 重构

1. 「给这段代码加注释，说明每个函数的作用」
2. 「把变量名改得更清晰」
3. 「把代码结构整理（顺序更合理）」
4. 「检查有没有性能问题并修复」

每次改动后测试，确保功能不变。
```

## M2-6 项目集成与接口设计（Week 5 Day 2）

### 理论逐字稿（30min）

```
今天学工程化的最后一关：前后端对接。

vibe coding 最常见的工程问题：
前端 AI 和后端 AI 生成的代码，接口格式对不上。

例子：
后端返回：{code: 200, data: {name: "张三"}}
前端期望：{status: "ok", result: {username: "张三"}}
两边格式完全不一样，前端永远拿不到数据。

正确做法：
第一步：先定义接口规范（让 AI 写 API 文档）
第二步：前后端各自按规范开发
第三步：对接时先测接口是否匹配

这样做的好处：
问题在源头就被堵住，而不是对接的时候才发现。
```

### 跟练任务（90min）

```
任务：做一个前后端分离的 TODO 应用

1. 先让 AI 写 API 文档：
「帮我写一个 TODO 应用的 API 文档，包含：
- GET /api/todos（获取列表）
- POST /api/todos（添加任务）
- DELETE /api/todos/:id（删除任务）
每个 API 说明：请求格式、响应格式、字段含义」

2. 基于文档生成后端（Express + Node.js）

3. 基于文档生成前端（HTML + JavaScript）

4. 对接测试：如果格式不对齐，记录问题并解决
```

## L4：综合项目（Week 6）

### Week 6 整体理念

```
L4 的目标：综合运用 L1-L3 的能力，完成一个完整项目。

项目要求：
1. 功能完整，可演示
2. 代码质量过得去（不是只有 demo 质量）
3. 有调试过程记录（不是一帆风顺做出来的）

答辩重点：
不是展示「代码多漂亮」
而是展示「遇到问题时的解决过程」
```

### 常见调试问题应对

| 问题 | 应对 |
|------|------|
| 报错信息看不懂 | 把报错发给 AI：「什么意思？」 |
| AI 修完还报错 | 让 AI 分析：「改了哪里？还可能有什么问题？」 |
| 一次修太多乱了 | 停下来，从头一个一个修 |
| 修完不确定对不对 | 必须跑一遍测试 |
| 对接格式不对 | 先查接口文档，再让 AI 对齐 |
