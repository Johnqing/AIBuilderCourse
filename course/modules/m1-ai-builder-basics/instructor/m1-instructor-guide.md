## 核心理念：产品经理的综合能力 × AI

```
产品经理的核心工作：

需求分析 → 方案设计 → 项目管理 → 数据汇报

每个环节都有大量重复性工作：
- 需求分析：反复解释同一个需求
- 方案设计：写文档、画原型、写PRD
- 项目管理：跟进度、发周报、催反馈
- 数据汇报：整理数据、发日报、异常预警

AI Builder能做的：
不是替代产品经理，而是把重复性工作自动化，
让产品经理把时间放在「需要人判断」的事情上。

M1的目标：
让产品经理学会用AI Builder，把自己的日常工作流程自动化。
```

## M1-1 产品架构入门 + AI辅助需求分析（Week 1 Day 1）

### 课前准备
- [ ] 准备产品经理日常工作场景案例
- [ ] 准备「AI辅助需求分析」演示
- [ ] 准备需求分析 Prompt 模板

### 理论逐字稿（30min）

```
今天进入M1。

先问大家一个问题：你们每天工作中，哪些事最费时间？

（等待2-3人分享：写PRD、跟项目、发周报、需求评审……）

好，我们把这些工作分个类：

第一类：重复性高但价值也高
→ 比如写PRD，每次都要写，但每次价值不大
→ 这类工作，AI可以帮你

第二类：重复性高且价值也高
→ 比如需求分析、跟进度，需要人判断
→ 这类工作，AI可以辅助你，但核心判断还是你

第三类：非重复性，高价值
→ 比如跟老板汇报、跨部门谈判
→ 这类工作必须你来做

M1的目标：
用AI Builder，把第一类工作自动化，
让AI辅助第二类工作，
让你有更多时间做第三类工作。

今天的核心：用AI辅助需求分析。

以前你们做需求分析，可能是开会、访谈、记笔记、整理成文档。
有了AI之后，你可以这样：

访谈时，让AI实时记录和总结；
访谈完，让AI帮你整理成结构化需求；
让AI帮你分析需求的可行性和优先级。

AI不是帮你做决策，而是帮你做「体力活」，
让你把精力放在「判断」上。
```

### 演示：AI辅助需求分析（20min）

```
演示场景：产品经理做用户访谈

1. 访谈时，打开飞书，让AI实时记录对话
2. 访谈结束，让AI总结关键需求
3. 让AI把需求整理成结构化格式
4. 让AI分析需求的优先级（MoSCoW）

讲解要点：
- AI做记录，人做判断
- AI整理结构，人做决策
- AI分析优先级，人做最终拍板
```

### 跟练任务（90min）

```
任务1：AI辅助访谈记录（30min）
1. 让学员模拟一段产品访谈
2. 用AI实时记录和总结
3. 对比：AI总结 vs 人工整理的差异

任务2：AI辅助需求整理（60min）
场景：用户反馈——"你们的APP打开太慢了，希望加点功能"

1. 把原始反馈发给AI，让AI分析：
   「分析这段用户反馈：
   1. 核心需求是什么？
   2. 背后可能的需求是什么？
   3. 如何转化为产品需求？」

2. 让AI用结构化格式整理需求

3. 让AI给出优先级建议（MoSCoW）
```

## M1-2 技能体系与选型 + AI辅助方案设计（Week 1 Day 2）

### 核心理念

```
今天的核心：用AI辅助写方案文档。

产品经理最费时间的工作之一：写PRD、写方案文档。
AI可以帮你：
1. 生成方案框架
2. 补充方案细节
3. 检查方案的完整性

但核心的「产品判断」——做什么、不做什么——还是人来做。
```

### 理论逐字稿（30min）

```
产品经理每天都在写文档：
PRD、MRD、需求文档、产品方案……

写得慢的原因：
1. 从零开始，不知道怎么组织结构
2. 细节要一个个想，效率低
3. 写完还要反复改格式

AI能帮你的是：
1. 快速生成文档框架
2. 补充常见的方案内容
3. 检查文档的完整性

但AI不能替你做的是：
1. 判断这个需求值不值得做
2. 确定优先级和资源
3. 跟老板和业务的博弈

今天的跟练：
让你体验用AI辅助写方案，把「体力活」交给AI，把「判断」留给自己。
```

### 跟练任务（90min）

```
任务：让AI辅助写PRD

场景：产品经理要做一个「智能客服」功能

1. 让AI生成PRD框架：
Prompt：「帮我写一个智能客服功能的PRD框架，包含：背景、目标、用户故事、功能列表、非功能需求、验收标准」

2. 让AI补充详细内容：
Prompt：「帮我补充智能客服的以下内容：
- 用户故事（3个典型用户）
- 功能详细描述（至少5个功能点）
- 异常流程说明」

3. 让AI检查PRD完整性：
Prompt：「检查这个PRD，告诉我还缺少什么」

4. 对比：用AI写 vs 自己从零写，花了多长时间，质量差多少
```

## M1-3 节点与路由 + AI辅助项目管理（Week 1 Day 3）

### 核心理念

```
今天的核心：用AI Builder搭建项目管理自动化工作流。

产品经理在项目管理上的痛点：
1. 每天要催进度，重复问同样的问题
2. 周报要手动整理，数据来源分散
3. 风险识别靠人盯，漏了没人知道

用AI Builder：
1. 定时提醒——AI自动发消息催进度
2. 自动周报——AI定时汇总项目数据
3. 异常预警——AI监控数据，达标就告警

这不是让AI替你管项目，而是让你的项目管理更高效。
```

### 理论逐字稿（30min）

```
产品经理管项目，最累的是什么？

每天早上问一遍：「昨天完成了什么？今天要做什么？有什么阻碍？」
每周写周报，把零散的信息整理成一篇文章。
每个风险都是人盯，盯漏了就是事故。

用AI Builder，这些都可以自动化：

场景1：每日站会提醒
→ AI定时发消息：「@XXX，今天的站会准备好了吗？请更新一下进度」
→ 产品经理只要回复，AI自动汇总

场景2：自动周报
→ AI每周五下午自动抓取各方的进度更新
→ AI生成周报草稿，产品经理修改后发出

场景3：风险预警
→ AI监控项目指标（进度延迟、bug超标）
→ 超过阈值自动发预警给负责人

产品经理的角色：
不是做「收集信息」的人，而是做「根据信息做决策」的人。
AI帮你收集，你来做判断。
```

### 跟练任务（90min）

```
任务：配置一个项目管理自动化工作流

1. 创建工作流「PM助手-姓名」
2. 配置定时触发：每周一到周五早上9点
3. 配置AI汇总节点：汇总各方的进度更新
4. 配置消息发送节点：把汇总发给项目群
5. 配置风险判断：进度delay>2天 → 发预警

场景测试：
手动触发工作流，看是否能正常汇总和发送消息
```

## M1-4 定时与触发 + AI辅助数据汇报（Week 1 Day 4）

### 核心理念

```
今天的核心：用AI Builder实现数据汇报自动化。

产品经理最不想做但又必须做的工作：
1. 日报/周报/阶段报
2. 数据监控和异常告警
3. 定期给领导的汇报材料

这些工作的特点：
重复、耗时、价值有限，但又不得不做。

用AI Builder自动化这些工作，
把产品经理从「信息搬运工」变成「分析决策者」。
```

### 理论逐字稿（30min）

```
产品经理的时间都花在哪了？

我见过最多的回答：
「每天写日报都要半小时」
「每周周报要整理一堆数据」
「每个月给老板汇报，要做PPT，数据还得一个个找」

这些事情有什么共同点？
重复、模板化、价值有限，但又不得不做。

用AI Builder，可以这样：

场景1：日报自动化
→ AI每天下午5点，自动从各系统拉数据
→ AI生成日报草稿，发给产品经理审核
→ 产品经理5分钟审核，发出

场景2：数据异常告警
→ AI定时监控核心指标（DAU、留存、转化）
→ 指标异常（下降>10%）→ AI自动预警

场景3：月度汇报辅助
→ AI帮你整理一个月的核心数据
→ AI生成汇报框架，产品经理填入洞察

核心逻辑：
AI做「信息收集和整理」
人做「分析和判断」
```

### 跟练任务（90min）

```
任务：配置一个数据汇报自动化工作流

1. 创建工作流「数据汇报-姓名」
2. 配置定时触发：每天下午5点
3. 配置数据拉取节点（模拟数据）
4. 配置AI生成节点：生成日报草稿
5. 配置消息发送节点：发送到飞书

测试：如果数据异常（比如转化率下降>10%），触发不同的告警消息
```

## M1-5 通道集成 + AI辅助跨部门协作（Week 1 Day 5）

### 核心理念

```
今天的核心：用AI Builder提升跨部门协作效率。

产品经理每天打交道的人：
UI、开发、测试、运营、老板……

协作痛点：
1. 同一个问题要重复解释多遍
2. 跨部门会议，记完没人看
3. 需求评审，要催很多遍反馈

用AI Builder：
1. 需求同步——一个消息自动发到多个群
2. 会议总结——自动生成会议纪要，发给相关人
3. 反馈收集——自动催收，自动汇总
```

### 理论逐字稿（30min）

```
产品经理是「信息的枢纽」。

每天要跟开发说需求、跟UI说设计、跟运营说计划、跟老板说进度。

问题来了：
1. 同一个信息，要发给不同的人，用不同的方式
→ 给开发的是详细技术方案
→ 给老板的是一句话结论

2. 开会的时候大家都在，但会后记不住
→ AI帮你录、帮你总结、帮你分发

3. 需求评审完，反馈收集要催很多遍
→ AI定时催、AI汇总、AI提醒

用AI Builder：
不是让你少沟通，而是让沟通更高效。
AI做「重复传达」的事，你做「关键判断」的事。
```

### 跟练任务（90min）

```
任务：配置一个跨部门协作自动化工作流

场景：需求评审会议后，自动生成纪要并发给相关人

1. 创建工作流「会议纪要-姓名」
2. 配置飞书会议记录触发（webhook）
3. 配置AI总结节点：生成会议纪要
4. 配置分支：
   - 有action items → 发送并@相关人
   - 无action items → 发送普通纪要
5. 配置提醒节点：action items的负责人确认

扩展：如果反馈超时未确认，发催确认消息
```

### 产品经理用AI Builder的checklist

```
上手前：
□ 明确要自动化哪个工作流
□ 确定AI处理的边界（什么让AI做，什么让人做）
□ 画出工作流的节点图

配置时：
□ 触发条件设置正确（定时/事件）
□ AI处理节点配置正确（用哪个技能）
□ 分支条件设置合理
□ 异常情况有兜底

上线后：
□ 测试通过
□ 定期检查AI输出质量
□ 根据反馈迭代优化
```
