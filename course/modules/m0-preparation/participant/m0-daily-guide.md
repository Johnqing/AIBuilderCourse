# M0 预备模块 学员每日指南

> Week 0-2（零基础版）/ Day 1-3（压缩版）：提示词工程核心
> **核心主题**：从「会用AI」升级到「用好AI」——结构化Prompt、错误预判、批判对话

## 核心理念：M0 不是「学用AI」，是「用好AI」

```
你已经会基本操作了——注册、会话、复制粘贴。

M0要教你的是：怎么用对方式问，得到可用的答案。

三个核心能力：
1. 结构化Prompt → 一次做对
2. 错误预判 → 堵住漏洞
3. 追问优化 → 持续改进
```

## Week 0

### Day 1-2：结构化Prompt

**今日目标**
- [ ] 掌握结构化Prompt公式（角色+任务+要求）
- [ ] 能区分低质量Prompt和高水平Prompt
- [ ] 能把工作中的常用任务写成Prompt模板

**结构化Prompt公式**

```
角色：你希望AI扮演什么角色？
任务：你要AI做什么？
要求：有什么具体要求？
  - 格式：JSON/纯文本/代码/列表
  - 受众：专业/通俗/老板/开发
  - 约束：不能做什么/必须做什么
  - 示例：有没有参考

示例：
「作为[资深产品经理]，帮我[写一份PRD]，要求：
- [包含背景、目标、用户故事、功能列表]
- [面向开发团队，语言专业]
- [参考附件模板风格]」
```

**今日任务**

```
跟练任务（上午）：
对比实验：同一任务，不同Prompt质量

任务：「帮我写一篇文章」vs「你是一个资深产品经理，帮我写一份产品需求文档，要求：包含背景、目标、用户故事、功能列表、验收标准，格式规范，语言专业，面向开发团队」

对比：两个答案的质量差异

独立练习（下午）：
从你的工作中选3个常用任务，写成结构化Prompt模板
```

### Day 3-5：错误预判与约束

**今日目标**
- [ ] 学会「错误预判」——在Prompt里堵住漏洞
- [ ] 掌握错误预判清单
- [ ] 能为复杂任务写防错Prompt

**错误预判清单**

```
写Prompt前，检查：
□ 边界条件：空值、超长输入、非法格式
□ 格式要求：JSON/纯文本/代码/表格
□ 受众限定：专业/通俗/管理层/执行层
□ 安全约束：不要生成什么
□ 完成标准：什么算完成
```

**今日任务**

```
跟练任务（上午）：
错误预判练习

场景：让AI生成用户注册表单

Step 1：低质量Prompt
「写一个用户注册表单」
记录AI出的错（字段不全？样式没有？验证缺失？）

Step 2：高质量Prompt（加约束）
「写一个用户注册表单，要求：
- 包含用户名、邮箱、密码、确认密码
- 密码至少8位，必须包含数字和字母
- 邮箱格式校验
- 有错误提示
- 提交前验证所有字段」

对比：高质量Prompt是否避开了之前的错误？

独立练习（下午）：
找一个你之前用AI时遇到的问题，分析：当初Prompt缺了什么约束？现在怎么补？」
```

## Week 1

### Day 6-8：批判对话与追问

**今日目标**
- [ ] 学会批判对话——不盲目接受AI的答案
- [ ] 掌握追问优化技巧
- [ ] 能通过多轮追问得到更精准的答案

**追问话术**

```
【质疑】
「你为什么这么说？」
「这个有依据吗？」

【深挖】
「具体指什么？」
「为什么会这样？」

【多角度】
「从另一个角度呢？」
「有没有反例？」

【验证】
「这个能落地吗？」
「执行步骤是什么？」
```

**今日任务**

```
跟练任务（上午）：
批判对话练习

1. 问AI一个有争议的问题
   「AI会取代产品经理吗？」

2. 追问：「你说"会取代"，具体指什么？」

3. 追问：「有什么数据支持这个观点？」

4. 追问：「有没有反面的观点？」

5. 追问：「如果我要应对AI取代，应该怎么做？」

对比：原始回答 vs 多轮追问后的回答质量差异

独立练习（下午）：
从工作中选一个真实问题，用多轮追问深入分析
记录：每轮追问的目的和效果
```

## Week 2

### Day 9-12：场景化Prompt

**今日目标**
- [ ] 掌握三个典型场景的Prompt模板
- [ ] 能综合运用结构化+错误预判+追问

**场景1：需求分析Prompt**

```
「你是一个资深产品经理，辅助我做需求分析。

背景：[用户反馈描述]

请帮我分析：
1. 这个反馈背后的真实需求是什么？
2. 可能的产品机会有哪些？
3. 如何转化为可落地的需求？

要求：
- 给出3个具体的产品机会
- 每个机会说明：目标用户、核心功能、预期价值
- 用结构化格式输出」
```

**场景2：方案设计Prompt**

```
「你是一个B端产品专家，帮我设计[系统类型]方案。

要求：
- [功能列表]
- [约束条件]
- [格式要求]

输出：
1. 整体架构
2. 核心功能模块
3. 实施计划
4. 技术注意点」
```

**场景3：代码生成Prompt**

```
「用[技术栈]写[功能]，要求：
1. [功能详细描述]
2. [约束条件]
3. [格式要求]

请分步骤：
1. [步骤1]
2. [步骤2]
3. [步骤3]」
```

## 提示词工程 Checklist

```
写Prompt前：
□ 我希望AI扮演什么角色？
□ 我要AI完成什么具体任务？
□ 输出格式是什么？
□ 受众是谁？
□ 有哪些约束条件？
□ 有什么不能犯的错误？（边界情况）

发出去之后：
□ AI理解对了吗？
□ 答案可用吗？
□ 有哪些地方需要追问？
□ 需要追问几次才能得到可用答案？

常用约束清单：
□ 格式：JSON/纯文本/代码/列表
□ 长度：不超过X字/X行
□ 语言：专业/通俗/中文/英文
□ 受众：管理层/执行层/开发/运营
□ 安全：不要生成敏感内容
□ 质量：要有数据支撑/要有具体例子
```

## 常见问题 FAQ

| 问题 | 答案 |
|------|------|
| 角色设定有用吗 | 有用，角色决定了AI的专业程度和表达风格 |
| Prompt越长越好吗 | 不是越长越好，是结构清晰、信息到位 |
| AI总是不按要求做 | 检查：约束条件写清楚了吗？有给示例吗？ |
| 同一个问题每次AI回答不一样 | 正常。可以加温度参数控制，或要求AI先复述你的要求 |
| 怎么判断AI回答对不对 | 不要盲目相信，用批判对话技巧质疑和验证 |
| 一个Prompt问几次算够 | 直到答案「可用」为止，不要在不可用的答案上凑合 |
